# إعادة هيكلة سيستم Vogatchi — تدقيق + خطة تنفيذ

## 1) الوضع الحالي (بيانات فعلية من قاعدة البيانات)

| العنصر | العدد | ملاحظة |
|---|---|---|
| الحجوزات (unified) | **183** | 155 فندق + 28 طيران |
| العملات | 144 EGP / 39 USD | مختلطة |
| الحالات | 178 confirmed / 5 pending | لا يوجد `completed` أو `cancelled` |
| الفواتير | 171 | 169 paid / 2 sent |
| مدفوعات موردين | 169 | مطابقة تقريبية للحجوزات |
| القيود المحاسبية | 384 | بيع + تكلفة + تحصيل + دفع مورد |
| العملاء | **692** | من ضمنهم 108 بدون هاتف، **691 بدون بريد** |
| Quotes | 0 | الموديول غير مُستخدم |
| جداول Legacy | 5 hotel_bookings فقط | باقي الأنواع اتنقلت |

### أهم المشاكل المكتشفة
1. **جدولة مزدوجة**: `hotel_bookings / flight_bookings / transport_bookings / car_rentals` موجودة كجداول legacy + views موحدة فوق `bookings`. الكود بيقرأ من views لكن ما زال في 5 صفوف hotel_bookings يتيمة.
2. **دورة الحياة ناقصة**: لا يوجد Quote → Booking، ولا حالات `completed / cancelled / refunded`، ولا Status History منتظم.
3. **جودة بيانات العملاء ضعيفة**: 99.8% بدون بريد، 15% بدون هاتف، احتمال تكرارات (نفس التليفون لأكثر من عميل).
4. **9 حجوزات بدون سعر بيع** و**18 حجز بربح سالب** (تكلفة > بيع).
5. **5 حجوزات بدون مورد** — بيكسر المحاسبة الخاصة بالتكلفة.
6. **العملاء المستوردين** غير مربوطين بمستخدم auth (كل الحجوزات باسم واحد).
7. **CRM فارغ**: لا leads، لا follow-ups، لا segments، لا communications log.
8. **المحاسبة**: القيود لكل عملة مستقلة (كويس)، لكن لا يوجد Trial Balance موحّد، ولا مطابقة بنكية.

## 2) خطة التنفيذ (5 مراحل — نبدأ فور موافقتك)

### Phase 1 — تنظيف وتوحيد البيانات (بدون فقد)
- ترحيل الـ 5 صفوف المتبقية من `hotel_bookings` إلى `bookings` + `booking_hotel_details`، ثم قفل الكتابة على الجداول القديمة (RLS: SELECT فقط للـ views).
- إضافة `deprecated_at` على الجداول القديمة والاحتفاظ بها للأرشيف.
- RPC `merge_duplicate_customers(phone)` لدمج العملاء المكررين مع نقل حجوزاتهم وفواتيرهم.
- تقرير Data Quality موسّع: العملاء بدون بريد، الحجوزات بربح سالب، الحجوزات بدون مورد، الفواتير غير المسددة.

### Phase 2 — دورة حياة الحجز الكاملة
- Enum موحد للحالات: `draft → quoted → confirmed → in_progress → completed → cancelled | refunded`.
- State machine على مستوى DB (trigger يمنع القفزات غير الشرعية) + `booking_status_history` تلقائي.
- ربط Quote ↔ Booking: عند تحويل Quote إلى Booking نسخ البنود وحفظ `source_quote_id`.
- عند `completed` تلقائياً: توليد فاتورة + قيد بيع + تحديث LTV العميل.
- عند `cancelled` تلقائياً: عكس القيود (reversing journal entry) + refund tracking.

### Phase 3 — المحاسبة والتقارير المالية
- **Trial Balance موحد** لكل عملة + سعر صرف تاريخي لتوليد Consolidated في EGP.
- **Cash Flow Statement** حقيقي (Operating / Investing / Financing).
- **AR Aging** (30/60/90) و**AP Aging** للموردين.
- **Bank Reconciliation**: مطابقة `bank_account_transactions` مع الفواتير/المدفوعات.
- إغلاق شهري نصف تلقائي (`accounting_periods` + قفل القيود).
- تصحيح الـ 18 حجز بربح سالب: تنبيه + مربّع تفسير (خصم/كومبليمنتري/خطأ إدخال).

### Phase 4 — CRM ومتابعة العملاء
- **Leads pipeline** كامل (New → Contacted → Proposal → Won/Lost) مع مصدر التسويق.
- **Follow-ups تلقائية**: تذكيرات قبل السفر، بعد السفر (طلب تقييم)، ذكرى الحجز السنوية.
- **Customer 360°**: تجميع كل التفاعلات (حجوزات، فواتير، رسائل واتساب، ملاحظات، تقييمات) في صفحة واحدة.
- **Segments ديناميكية**: VIP (LTV > X)، متكرر (>3 حجوزات)، خامل (>6 أشهر بدون حجز).
- استيراد بريد العميل تلقائياً من بيانات الفاتورة/الجواز.

### Phase 5 — تكامل الموديولات + التحقق
- **Booking 360°**: كل حجز يعرض العميل + المورد + الفاتورة + المدفوعات + القيود + المحادثات في مكان واحد.
- **Supplier Statement**: كشف حساب مورد PDF (المستحقات + المدفوعات + الرصيد).
- **Customer Statement**: كشف حساب عميل PDF.
- Automation: عند إنشاء حجز → إشعار واتساب للعميل + مهمة متابعة للموظف.
- اختبار End-to-End على بيانات فوجاتشي الحقيقية والتأكد من تطابق كل الأرقام قبل/بعد.

---

## 3) ضمانات السلامة
- كل مرحلة migration واحد قابل للـ rollback + snapshot للجداول المتأثرة قبل التعديل.
- لن يُحذف أي صف — فقط ترحيل أو تحديث.
- عرض تقرير قبل/بعد لكل مرحلة (عدد الصفوف، مجاميع مالية) عشان تتأكد.

---

## 4) الترتيب المقترح للبدء
أبدأ بـ **Phase 1** (تنظيف + تقرير جودة موسّع) لأنها تكشف أي مشاكل خفية قبل تعديل المنطق. تقديرياً migration واحد + hook واحد + صفحة تقرير.

هل نبدأ بـ Phase 1 مباشرة، أم تحب أعرض تقرير جودة تفصيلي بالأرقام لكل بند قبل التعديل؟
